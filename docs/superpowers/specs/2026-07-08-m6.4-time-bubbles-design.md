# Milestone 6.4 — Time Bubbles: Design

*Status: DRAFT for review (2026-07-08). Final milestone-6 slice, and the most
architecturally significant — it changes the core simulation from one global
clock to **local time domains**. Builds on 6.1 (phased move), 6.2 (monsters),
6.3 (combat). Rebase this branch onto `main` once #16 (6.3) merges.*

## Goal

The headline mechanic: **when a player and a monster come within combat range,
the clock freezes *locally* for them and anyone nearby, and the fight becomes a
deliberate, action-gated turn-based battle — while the rest of the world keeps
running at the normal 5-second WeGo pace.** Friends elsewhere see the fight
frozen mid-swing ("in combat") and can **walk into it to join**; break away and
you slip back into normal time.

Per the scope decision (2026-07-08): **the whole mechanic in one slice**, using
**distance-based** detection (hex distance within `CombatRadius = 6`). Terrain-
blocked line-of-sight is **deferred** to a follow-up.

## The core model: time domains

Today every entity lives in one global clock (`resolveTurn` every
`TURN_INTERVAL`). 6.4 splits the world into **domains**:

- **The world domain** — everyone not in a fight. Ticks every `TURN_INTERVAL`
  (5s WeGo), resolving movement with the 6.1 phased move (monsters hunt, players
  walk). This is the existing loop, now scoped to world-domain entities only.
- **Combat bubbles** — each a set of entities frozen in local combat time. A
  bubble advances **action-gated** (see below), *not* on the world clock.

Every entity is in exactly one domain at any instant: the world, or one bubble.

### Bubbles as connected components (form / grow / merge / dissolve — all one rule)

Rather than bespoke form/merge/absorb logic, bubbles are **recomputed as
connected components** of a proximity graph, each time positions change (after
every world tick and every bubble resolution):

1. Build a graph over all entities; connect two entities if `HexDistance <=
   CombatRadius`.
2. A connected component is a **bubble** iff it contains at least one **opposing
   pair** (a player and a monster). *Every* entity in such a component — including
   nearby friendlies — is a bubble member (that's "anyone standing nearby joins").
3. A component with no opposing pair is **world-domain** (a lone player, a pack
   of monsters with no player in range, etc.).

This single rule yields the whole lifecycle for free:
- **Form**: a player and monster come within range → their component gains an
  opposing pair → becomes a bubble.
- **Grow / walk-in reinforcement**: a world entity steps within `CombatRadius`
  of a bubble member → next recompute it's in the same component → absorbed.
- **Merge**: two bubbles drift within range → one component → one bubble.
- **Dissolve / escape**: the last monster dies, or a player moves out of range so
  their sub-component no longer has an opposing pair → back to the world domain.

**Bubble identity across recomputes:** to keep a bubble's action-gating state
(who's locked in, the patience deadline) stable turn-to-turn, a recomputed
component inherits the id of the prior bubble it most overlaps (carry the
`pendingIntents`/deadline forward); genuinely new bubbles get a fresh id; a
merge keeps one id and folds the others' members in.

## The two clocks

### World domain — unchanged cadence
A `TURN_INTERVAL` ticker resolves the world domain: the 6.1/6.3 phased move for
world-domain entities only (world monsters hunt the nearest **world** player;
world players walk). Then recompute bubbles (movement may have created one).

### Bubble — action-gated local clock
A bubble resolves one combat turn when **either**:
- **all member players have locked in** an intent for this bubble-turn (monsters
  are always ready — their AI produces an intent at resolution), **or**
- the **patience timeout** elapses (`COMBAT_PATIENCE`, env-configurable, default
  **60s**; shrunk in tests). Players who didn't lock in simply wait (no move).

On resolution: gather intents (locked-in players + monster AI), run the **6.3
combat `resolveTurn` logic on the bubble's members only** (faction-aware move →
simultaneous attack → damage → deaths/respawn), then recompute bubbles
(dissolve/merge/absorb), reset lock-ins, restart the patience timer, and publish.

**Escape** falls out of resolution + recompute: a player who moves out of range
this bubble-turn leaves the bubble on the recompute.

### Driving both without per-bubble goroutines
`Run` uses one control loop on a small poll interval (a new `BUBBLE_POLL`,
default ~100ms). Each poll: (a) if a full `TURN_INTERVAL` has elapsed → resolve
the world domain; (b) for each bubble, if all-ready **or** past its deadline →
resolve it. `SubmitIntent` additionally resolves a bubble **immediately** when
its last player locks in (snappy), guarded so the poll's timeout path can't
double-resolve. All state transitions hold `w.mu`.

## Wire changes (`internal/protocol`)

- **`Entity.InCombat bool`** — true when the entity is in a bubble. Lets every
  client render frozen fighters as "in combat", and lets my client switch to
  combat mode when *my* entity is in a bubble.
- **Bubble view for the acting client.** The turn bundle gains
  `Bubbles []BubbleView`, each `{ID int64; MemberIDs []int64; WaitingForIDs
  []int64; PatienceRemainingMs int64}`. The client finds the bubble containing
  its own entity to drive the "waiting for…" HUD; other bubbles are ignorable.
  (Bubbles are few — this stays small.)
- **Event ids.** With domains advancing independently, SSE ids become a global
  monotonic **event sequence** (increment per publish) rather than the world
  turn number; `Last-Event-ID` still works as a resume watermark. The world
  `turn` stays in the bundle for the world-domain HUD.

## Client (`client/`)

- **In-combat rendering:** entities with `InCombat` get a distinct marker (e.g.
  a combat ring / frozen tint) so world players *see* frozen fights and can walk
  in. Reuse the existing dot/HP-bar layer.
- **Combat mode (my entity is in a bubble):** the turn-timer HUD switches to a
  **combat panel** — "Waiting for: `<names/ids>` · `Ns` left" (from
  `PatienceRemainingMs`), and my movement intent reads as a locked-in combat
  action. When not in a bubble: the existing WeGo turn timer.
- **`window.game`** exposes `inCombat: boolean`, and `bubble: {waitingFor:
  number[]; patienceRemainingMs: number} | null` for e2e.

## Config (`internal/config`)
- `COMBAT_PATIENCE` (duration, default `60s`) — bubble patience timeout.
- `BUBBLE_POLL` (duration, default `100ms`) — the control-loop poll interval.
Both env-configurable so tests/e2e can shrink them (like `TURN_INTERVAL`).

## Decisions to review (please weigh in)

1. **Distance-based bubbles as connected components** (your scope call: distance,
   not terrain LOS). The connected-component rule is my proposal for form/merge/
   grow/dissolve as one mechanism — simpler and fewer edge cases than bespoke
   join/leave logic. OK?
2. **Action-gating semantics:** resolve on *all players locked in* OR patience
   timeout; AFK/unready players just wait (no move) — they're not skipped from
   the fight, only from that turn's action. Patience default 60s, env-tunable.
3. **Reinforcement = same connected component.** Walking within `CombatRadius`
   of any bubble member pulls you in (no invite). Matches the design's "step into
   it". OK?
4. **Monsters are always ready** (AI intent at resolution) so a bubble only ever
   waits on *players*. OK?
5. **Wire shape:** per-entity `InCombat` + a small `Bubbles` list in the bundle +
   switch SSE ids to a global event sequence. Alternative: a leaner
   `InCombat`-only wire with the "waiting" info derived client-side (less
   precise). I recommend the explicit `Bubbles` view.
6. **Respawn on death inside a bubble:** a player who dies respawns in the
   **world domain** at a spawn hex (leaves the fight), per 6.3 respawn. OK?

## Out of scope (deferred)

- **Terrain-blocked line-of-sight** (rock/forest blocks "spotting") — a follow-up;
  distance-only here.
- **XP / shared-XP-on-kill / classes / ranged / AoE** — 6b.
- **Monster respawn / continuous spawning** — later tuning.
- **Rich combat UI** (initiative order, ability bars) — this slice ships a
  functional "waiting for… / Ns" panel, not a polished combat HUD.
- **Per-bubble reconnect nuance** — the existing resync-to-latest model applies;
  a reconnecting client gets the current snapshot incl. its `InCombat`/bubble
  state.

## Tests

- **Unit (`internal/game`)**: bubble **forms** when a player and monster close to
  `<= CombatRadius` (both flip `InCombat`); a nearby friendly is **absorbed**;
  the bubble does **not** advance until the player locks in (world ticks pass, the
  bubble's members don't move); it **does** advance once locked in (combat turn
  runs); **patience timeout** advances it with an unready player waiting; a
  world entity **walking into range joins**; killing the last monster
  **dissolves** the bubble (members return to world domain); a player moving out
  of range **escapes**; two bubbles **merge**. Pin seeds; use a tiny
  `COMBAT_PATIENCE` and drive resolution deterministically via test bridges.
- **Integration (`test/integration`)**: over real HTTP/SSE, a player walking into
  a monster enters combat (`InCombat` true in the bundle, bubble view present);
  the fight doesn't advance until the player posts an intent; a second player
  walking in joins the same bubble.
- **e2e (`client/e2e/combat.spec.ts` extension or a new `*combat.spec.ts`)**:
  driving the player into a monster shows the combat panel / `window.game.inCombat`
  flips; the world keeps ticking (world `turn` advances while the bubble waits).
  Runs on the combat server.

## Risks

- **Concurrency**: multiple resolution triggers (world ticker, bubble poll,
  `SubmitIntent` immediate-resolve, patience path) must all serialize on `w.mu`
  and not double-resolve a bubble-turn. Define a per-bubble "resolving" guard /
  turn counter.
- **Bubble identity across recomputes** must be stable or the "waiting for" set
  and patience deadline flicker. The most-overlap inheritance rule needs care.
- **The wire/event-id change** (turn-number → event sequence) touches the SSE
  layer and `Last-Event-ID` resume — keep the watermark semantics intact and
  covered by the existing reconnect tests.
- **Scope**: this is the largest slice; the plan below decomposes it into
  independently-reviewable tasks (domain model → formation → gating/loop → wire →
  client → config → tests). If review finds it too big, the natural cut line is
  after the server mechanic (domains + gating) vs the client HUD.
- **Depends on 6.3 (#16)**: combat `resolveTurn`, factions, HP, death/respawn —
  rebase onto `main` after #16 merges before building.
