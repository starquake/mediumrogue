# Milestone 8.1 — Chat: Design

*Status: DRAFT (2026-07-09). First slice of milestone 8 (chat / parties /
quests). Ships a global chat channel, the project's first **SolidJS** component,
a display **name** chosen at join, and a server-parsed **`/` command** framework
with one command (`/here`, share location). Branch off `main`.*

## Goal

Give the ~15-friend group the social loop the whole design leans on ("help, 3
ghouls, north bridge!" → cavalry arrives): a **global** in-game chat, delivered
live over the existing SSE stream, typed into a DOM panel over the Pixi canvas.
This slice also stands up the **SolidJS** toolchain (decided in §9) as the first
reactive-DOM component, and adds the **name** half of the plan's "name + secret
link" identity.

Decided forks (user, 2026-07-09):
1. **Three slices** for milestone 8 — **8.1 chat** (this), 8.2 parties, 8.3
   quests — built in dependency order.
2. **Global channel only** (party/local channels come with 8.2).
3. **Ephemeral** — a client sees only messages sent while it is connected; no
   history buffer, no replay on reconnect, nothing persisted.
4. **Display name at join** — the chat sender label.
5. **`/` command framework, server-parsed**, with **`/here`** (share location).

## Chat delivery — a non-coalescing broker

The turn hub (`internal/hub`) **coalesces** (`struct{}` wake-ups, "fetch latest
state"), which is exactly wrong for chat: discrete messages must each arrive.
So chat gets its own fan-out.

New package `internal/chat`:
```go
// Broker fans out chat messages to in-process subscribers. Unlike the turn
// hub it does NOT coalesce — each message is delivered (best-effort). A slow
// subscriber whose buffer is full drops messages rather than blocking Publish
// (ephemeral: no delivery guarantee, no history).
type Broker struct { … }

func NewBroker() *Broker
// Subscribe returns a buffered receive channel (cap chatBufferN, e.g. 32) and
// an unsubscribe func (defer it). Publish assigns a monotonic Seq.
func (b *Broker) Subscribe() (<-chan protocol.ChatMessage, func())
func (b *Broker) Publish(sender, text string)  // stamps Seq, fans out
```
- `Seq` is a monotonic `atomic.Int64` — a stable client key + ordering aid. **No
  wall clock** (test-safe, deterministic ordering).
- Buffer-full → drop for that subscriber only (best-effort matches "ephemeral").
- Mirrors the hub's Subscribe/unsubscribe shape and concurrency-safety.

## SSE + POST wiring

- **`handleEvents`** (`internal/server/events.go`) gains a `case msg :=
  <-chatCh:` in its select loop that writes an **`event: chat`** frame with the
  JSON `ChatMessage` as data and **no `id:`** (chat must not advance
  `Last-Event-ID`, same rule as `heartbeat`). It interleaves with turn and
  heartbeat frames on the one stream. The handler `Subscribe()`s to the broker
  after the turn subscription and `defer`s unsubscribe.
- **`POST /api/chat`** (new route + `handleChat`): body `ChatRequest{Token,
  Text}`. Flow:
  1. Resolve the sender from the token via a new locked world read
     `SenderFor(token) (name string, hex protocol.Hex, ok bool)`. `!ok` (unknown
     token / not joined) → **401**.
  2. `Text = strings.TrimSpace(req.Text)`; empty → **422**; longer than
     `MaxChatLen` (≈500 runes) → **422**.
  3. If `Text` starts with `/`, dispatch to the **command registry** (below).
     Otherwise the message body is `Text`.
  4. `Broker.Publish(name, finalText)`.
- `Deps` gains `Chat *chat.Broker`; `app.go` constructs one broker and passes it
  to both `New(deps)` (for the POST handler) and the SSE handler via `deps`.

### Command framework (`/…`)

Server-parsed so location can't be spoofed (the server supplies coordinates from
the authoritative entity position).

- Parse: `verb, args := firstToken(text[1:]), rest`. A `map[string]cmdFunc`
  registry; a `cmdFunc(ctx cmdContext) (broadcastText string, err error)` where
  `cmdContext` carries the sender's `name` and `hex`.
- **`/here`** → `broadcastText = "📍 at (q, r)"` from the sender's hex; broadcast
  as a normal `ChatMessage` (sender = the player's name), so it renders in every
  panel and the coordinates are assertable in tests.
- **Unknown verb** → `ErrUnknownCommand` → **422** with an error code/message;
  **not** broadcast. The client renders a **local-only** system line
  ("Unknown command: /foo") so it doesn't spam the channel.
- The registry is the extension point for 8.2/8.3 (`/invite`, `/quest`, …);
  `/here` is the only command this slice ships.

### XSS / safety

Chat text is untrusted. SolidJS renders it as **text nodes** (not `innerHTML`),
and the CSP forbids `unsafe-eval`/inline — so injection is covered by the
framework + CSP. The server only **trims** and **length-caps**; it does not
strip/escape HTML (the client never interprets it as markup).

## Wire (`internal/protocol`) — regen `protocol.gen.ts`

- `ChatMessage{ Seq int64 json:"seq"; Sender string json:"sender"; Text string json:"text" }`
- `ChatRequest{ Token string json:"token"; Text string json:"text" }`
- `EventChat = "chat"` (SSE event-name constant, beside `EventTurn`/`EventHeartbeat`).
- `MaxChatLen` constant (rune cap, e.g. 500).
- `JoinRequest.Name string json:"name"` — **required** for a new player
  (non-empty after trim, ≤ `MaxNameLen` e.g. 24 runes; ignored on reclaim like
  class/species). Invalid → new `ErrInvalidName` → **422**.
- `Entity.Name string json:"name"` — exposed in the turn snapshot (lets the
  client label itself now; parties/quests reuse it). Set from `entity.name`.

`Join`'s signature becomes `Join(token, name, class, species string)`; all
callers (production `api.go`, tests) pass a valid name. `entity` gains `name`.

## Client (`client/`) — SolidJS + `<ChatPanel>`

**Stand up SolidJS** (first component; §9 decision):
- Add deps `solid-js` + dev `vite-plugin-solid`; register the plugin in
  `vite.config.ts`. `tsconfig.json`: `"jsx": "preserve"`,
  `"jsxImportSource": "solid-js"` so `tsc --noEmit` type-checks the TSX (the
  deciding "checkable in one tool" criterion). Compile-time → **CSP-safe** (no
  `new Function`/eval); **light-DOM** output (global CRT theme + Playwright
  selectors); bundle still `go:embed`-ed via `internal/web`.

**`<ChatPanel>`** — a DOM overlay beside the HUD:
- A scrollable message list (`for` over a `messages` signal, keyed by `Seq`) +
  a text `<input>` + send button. Sender name in a distinct style; auto-scroll
  to newest.
- Mounted into a container element in `index.html` (e.g. `#chat-root`), styled
  to float over the canvas and **take pointer events only on itself** (so map
  clicks elsewhere still reach Pixi).
- The SSE `event: chat` handler (in `net/events.ts`) parses `ChatMessage` and
  appends to `messages` (cap the retained list, e.g. last 200, purely client
  memory hygiene — not "history").
- Send: `POST /api/chat {token, text}`; on **422** command error, append a
  **local-only** system line to `messages`; clear the input on success.

**Name at join:** a name `<input>` in the join UI (beside the class/species
pickers), default/placeholder so existing e2e still auto-join with a default
name. `join(name, class, species)`.

**`window.game` (Playwright surface):**
- `name: string` (my display name), `chat: { seq: number; sender: string; text: string }[]`
  (received + local system lines, in order), `sendChat(text: string): Promise<void>`.

## Decisions (reasoned; flag if you disagree)

1. **Server-parsed commands** (not client) — location must be authoritative
   (anti-spoof) and the registry is server-side extensible for 8.2/8.3.
2. **Location rides as plain `ChatMessage` text** (`📍 at (q, r)`) — no extra
   wire field this slice; coordinates stay assertable. Clickable/actionable
   location (focus camera on the hex) is deferred polish.
3. **Ephemeral, best-effort delivery** — no history, no per-message ack; a
   buffer-full slow client drops messages (matches the chosen model; simplest
   broker).
4. **`Seq` is a broker atomic**, not a timestamp — deterministic, no clock.
5. **Name required at join** — the plan's identity is "name + secret link";
   chat needs a sender; parties/quests need it too.

## Out of scope (later)

- **Party / local / proximity channels** (8.2); **quest-pitch `/` commands**
  (8.3, e.g. `/invite`, `/quest`).
- **History / persistence / replay on reconnect**; **delivery guarantees**.
- **DMs, mentions, moderation, emotes, rate-limiting** (a simple length cap
  only for now).
- **Clickable shared-location** (focus/pathfind to the shared hex).
- **Renaming / name uniqueness** (names need not be unique in this slice).

## Tests

- **Unit (`internal/chat`)**: `Publish` fans out to N subscribers in order;
  `Seq` strictly increases; a full subscriber buffer drops for that subscriber
  only (others still receive); unsubscribe stops delivery and doesn't block
  `Publish`. **Command parser**: `/here` formats the sender's hex; an unknown
  verb returns `ErrUnknownCommand`; a bare `/` / whitespace is handled.
- **Integration (`test/integration`)**: two SSE clients on one server; client A
  `POST /api/chat` a plain message → client B receives an `event: chat` frame
  with A's **name** + text. A `POST /api/chat "/here"` → B receives a message
  containing A's name and A's **coordinates**. Empty text → **422**; oversize
  text → **422**; unknown token → **401**; `/bogus` → **422**. `JoinRequest`
  without a name → **422**.
- **e2e (`client/e2e/chat.spec.ts`)**: two browsers; A types a message and
  sends → it appears in **B's** chat panel (and `window.game.chat`) with A's
  name. A types **`/here`** → a location line with coordinates appears. The
  name-at-join flow works and both clients still auto-join (default name). Its
  own Playwright server (added to the `specs` list).

## Risks

- **Coalescing trap**: chat must NOT ride the turn hub (it would drop messages).
  The dedicated broker + a distinct SSE `case` is the guard; the integration
  test (two messages, both received) proves it.
- **SolidJS toolchain**: the first Solid build must keep `tsc --noEmit` green
  (jsx settings), stay CSP-safe (no eval — Solid is compile-time), and embed
  cleanly. Verify `make check` (tsc + build + go:embed) and the CSP headers.
- **Pointer capture**: the chat DOM must not swallow map clicks — scope
  pointer-events to the panel, and keep an e2e that a map click still moves the
  player with the panel present.
- **Name-at-join wiring**: two-field-plus-name join must still auto-join with
  defaults so existing e2e don't hang (same discipline as the class/species
  pickers).
