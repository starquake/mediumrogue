# Milestone 8.2 — Parties: Design

*Status: DRAFT (2026-07-09). Second slice of milestone 8 (chat / parties /
quests). Invite-based party membership with a roster and on-map distinction —
the scaffolding 8.3 party-quests build on. Branch off `main`.*

## Goal

Let players form parties by invitation, see their party in a roster, and pick
their partymates out on the map. No shared movement or combat perks yet — this
slice is membership + visibility. It also gives the `/`-command registry (built
in 8.1) its first *state-mutating* commands and adds the project's **second**
SolidJS component (`<RosterPanel>`).

Decided forks (user, 2026-07-09):
1. **Invite + accept handshake** (`/invite <name>` announced, target `/accept`s)
   — not open-join.
2. **Scope = membership + roster + map distinction** — no shared-party movement,
   no combat perks this slice.

Decided defaults (stated, not objected to):
- **Flat** structure (any member can invite; no leader/kick).
- Party events announced in **global chat** (reuses the chat rails).
- A party **persists across death/respawn**; **dissolves** when it drops below 2
  members; a **disconnect-sweep** removes you from it.

## Party model (game layer, `internal/game/world.go` + `party.go`)

- `entity.partyID int64` (0 = solo). A **party is ≥2 players** sharing a
  non-zero id. `Entity.PartyID` rides the turn snapshot.
- Server state: `w.pendingInvites map[int64]int64` (invitee entity id → inviter
  entity id); `w.nextPartyID int64`.
- Operations are **locked `World` methods**, each returning an **announce
  string** (broadcast) or a sentinel error (→ 422). They resolve the caller from
  their token:
  - `PartyInvite(token, targetName string) (announce string, err error)` —
    resolve the inviter; resolve the target by **name** among players (exact
    match; if a name repeats, the **nearest** player to the inviter,
    lowest-id tie-break); errors: `ErrTargetNotFound` (no such name),
    `ErrInviteSelf` (only match is yourself). Record `pendingInvites[target]
    = inviter`. Announce: `"alice invited bob to a party — bob: /accept"`.
  - `PartyAccept(token string) (announce, err)` — require a pending invite
    (`ErrNoPendingInvite`); the inviter must still be present
    (`ErrInviteExpired`); if already in the inviter's party
    (`ErrAlreadyInParty`). Otherwise: leave your current party (dissolve if it
    drops <2), then join the inviter's party (creating a new party id with both
    if the inviter had none). Clear the pending invite. Announce:
    `"bob joined alice's party"`.
  - `PartyLeave(token string) (announce, err)` — `ErrNotInParty` if solo; else
    leave and dissolve the remainder if <2. Announce: `"bob left the party"`.
- Shared helper `leavePartyLocked(e *entity)` (used by Leave, Accept's
  old-party exit, and the disconnect sweep): clears `e.partyID`, and if the old
  party now has <2 members, clears the lone survivor too (dissolve).
- **Disconnect sweep** (`sweepDisconnectedLocked`): before deleting a gone
  entity, `leavePartyLocked(e)` and purge it from `pendingInvites` (as invitee
  key AND as any inviter value).
- `Snapshot` sets `Entity.PartyID = e.partyID`. Monsters: always 0.

## Command routing (`internal/server/chat.go`)

`handleChat` parses the verb of a `/`-message and routes:
- `invite` / `accept` / `leave` → the matching `World` method; on success
  **broadcast the announce under a `system` sender** (so it reads as a neutral
  announcement to everyone, and the client's existing `sender === "system"`
  styling applies); on a sentinel error → **422** (surfaced as a local system
  line, exactly like `/bogus` today).
- anything else (e.g. `/here`) → the existing pure `chat.RunCommand`, broadcast
  under the **player's** name (unchanged).
- The verb split is a tiny local helper (`cutVerb(text) → verb, rest`); the
  target name is the whole remainder (so names with spaces work).

No `/party` command — the roster panel **is** the party view.

*(Note: `"system"` is not a reserved name yet; a player could in theory name
themselves `system`. Reserving it is a cheap follow-up, out of scope here.)*

## Wire (`internal/protocol`) — regen `protocol.gen.ts`

- `Entity.PartyID int64 json:"partyId"` — the only wire change. Roster and
  map-distinction are **derived client-side** by grouping entities on `PartyID`.
- **No new SSE events** (party state rides the turn bundle; announcements ride
  chat) and **no new request types** (commands ride `POST /api/chat`).

## Client (`client/`)

**Roster (`<RosterPanel>` — second Solid component):**
- `client/src/party/store.ts`: a `party` signal (the member **names** of my
  party, including me), updated each turn from the bundle; `partyId` too.
- `client/src/party/RosterPanel.tsx`: "Party" header + `<For>` over the names;
  hidden/empty when solo. `mountRoster(root)` (JSX stays in the `.tsx`).
- Mounted into a `#roster-root` overlay in `index.html` (same pointer-events
  discipline as chat: `#roster-root` click-through, panel captures its own).

**Map distinction (`client/src/render/entities.ts`):**
- Add `PARTY_COLOR`. `EntityLayer.update(entities, myEntityID, myPartyID,
  playbackMs)` marks a dot a **partymate** when
  `e.partyId !== 0 && e.partyId === myPartyID && e.id !== myEntityID`, colored
  `PARTY_COLOR` (priority: hostile → mine → partymate → other).
- `main.ts` passes `mine.partyId` (or 0) into `update` and refreshes the party
  store each turn.

**`window.game` (Playwright surface):** `party: string[]` (my party's member
names) and `partyId: number`.

## Decisions (reasoned; flag if you disagree)

1. **Party state = just `Entity.PartyID`** on the wire; roster/coloring are
   client-derived. Minimal wire, no new event/request. A dedicated `PartyView`
   list would be redundant at this player count.
2. **Announcements as a `system` sender over global chat** — reuses 8.1's rails
   and the existing system-line styling; everyone sees parties form (the
   intended social texture).
3. **Party = ≥2**; a lone member is just solo (`partyID = 0`). Avoids "a party
   of one" edge state.
4. **Invite by name, nearest-match on collision** — forgiving for a friend group
   (names aren't unique); deterministic tie-break.
5. **Party commands are `World` methods** (game-layer state), routed by the chat
   handler; `/here` stays a pure `chat` command. Party logic is unit-testable in
   the game package.

## Out of scope (later)

- **Shared-party movement** (one destination moves the blob), **party combat
  perks**, **leader/kick**, a **party chat channel**, **invite expiry/timeout**,
  reserving the `system` name, and **party-quests** (8.3, which adds `/quest`-style
  commands on this same registry + membership).

## Tests

- **Unit (`internal/game`)**: invite records a pending; accept forms a shared
  non-zero party (both ids equal); accept into an inviter's existing party adds
  a third; leave from a party of 3 leaves the other two together; leave from a
  party of 2 dissolves both to solo; disconnect-sweep removes + dissolves and
  purges pending invites; `ErrNoPendingInvite` / `ErrTargetNotFound` /
  `ErrInviteSelf` / `ErrInviteExpired` / `ErrAlreadyInParty` / `ErrNotInParty`;
  nearest-match on duplicate names; `Snapshot` PartyID. Pin any determinism.
- **Integration (`test/integration`)**: over HTTP + chat — alice `/invite bob`,
  bob `/accept` → both `/api/events` bundles show the same non-zero `PartyID`;
  bob `/leave` → 0; `/invite ghost` → 422; the announcements arrive as
  `event: chat` frames with sender `system`.
- **e2e (`client/e2e/parties.spec.ts`)**: two browsers — A `/invite <B name>`,
  B `/accept` → both `window.game.party` contain both names and `partyId` is
  equal + non-zero; both roster panels (`#roster-*`) show both names; B `/leave`
  → rosters clear and `partyId` → 0. (Map color is a visual nicety, covered by
  the unit/integration `PartyID` correctness rather than a brittle canvas
  assertion.) Its own Playwright server (added to the `specs` list).

## Risks

- **Command routing split**: party verbs must route to `World` BEFORE the pure
  `chat.RunCommand` default, or they'd 422 as unknown. The integration test
  (invite/accept actually forming a party) guards it.
- **Dissolve correctness**: leaving/disconnect must dissolve a party that drops
  below 2 (no orphan party-of-one). Unit tests cover 2→dissolve and 3→2-stay.
- **Name collisions**: `/invite <name>` nearest-match must be deterministic
  (lowest-id tie-break) so tests and behavior are stable.
- **Sweep ordering**: purge `pendingInvites` (both directions) and leave the
  party when an entity is swept, or a stale accept could target a ghost — the
  `ErrInviteExpired` check is the backstop, but purge anyway.
