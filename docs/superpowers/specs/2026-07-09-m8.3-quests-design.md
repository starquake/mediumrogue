# Milestone 8.3 — Quests: Design

*Status: DRAFT (2026-07-09). Final slice of milestone 8 (chat / parties /
quests). A seeded server quest board with kill-count and reach-hex objectives,
XP rewards, party-shared progress, and a SolidJS quest log. Branch off `main`.*

## Goal

Give the group things to *do*: a small board of server-generated quests a
player (or party) takes, progresses, and completes for **XP** — the loop the
plan's "pitch it in chat, who's coming?" social texture hangs off. Completes
**milestone 8** (8.1 chat, 8.2 parties, 8.3 quests).

Decided forks (user, 2026-07-09):
1. **Server quest board** — seeded, generated at boot (no NPCs, no authoring).
2. **Two objective types** — **kill-count** (slay N monsters) and **reach-hex**
   (stand on a goal hex).
3. **Party quests pay everyone present at completion, in full** — progress is
   pooled party-wide; whoever is in the party when it completes gets the whole
   reward (the XP "walking in to help pays" philosophy; §9 decision).
4. **One active quest per player** — your slot is your personal quest OR your
   party's quest (§9 decision).

Derived one-slot rules (stated in brainstorm, approved):
- Taking a quest while you (or your party) already hold one → readable 422.
- **Joining a party abandons your personal quest** (announced) — your slot is
  the party's agenda while you're in it. Leaving the party leaves the quest
  with the party (your slot empties).

User requirement: **XP must be visible in the UI** — the existing stats HUD
(`Lv X · Y/Z XP`) already renders XP; quest completion must visibly bump it,
the quest panel shows each quest's XP reward, and the e2e asserts the XP jump
on completion.

## Quest model (game layer)

New file `internal/game/quest.go` (+ generation in `questgen.go` or same file):

```go
type questKind string
const (questKill questKind = "kill"; questReach questKind = "reach")

type quest struct {
	id       int64
	name     string
	kind     questKind
	targetN  int          // kill: monsters to slay
	goalHex  protocol.Hex // reach: the hex to stand on
	rewardXP int
	state    protocol.QuestState // available | taken | completed
	progress int
	// exactly one holder when taken:
	holderEntity int64 // personal quest (0 otherwise)
	holderParty  int64 // party quest (0 otherwise)
}
```

**Board generation** — `generateQuests(seed uint64, m protocol.MapResponse) []*quest`,
called once in `NewWorld` (seeded from `WORLD_SEED` → deterministic, like the
map): **6 quests** — 3 kill (counts 2–4; modest because monsters don't respawn)
and 3 reach. Reach goal hexes are drawn from the **origin-reachable walkable
region** (`reachableWalkable(m)` — guaranteed reachable), at a meaningful
distance from the origin (e.g. ≥ 8 hexes), picked with a seeded hash/PCG —
**no `math/rand` global, no clock**. Names from small templates
("Cull the pack", "Scout the far shore", …).

**Operations** (locked World methods; announce string or sentinel → 422):
- `QuestTake(token, questID) (announce, err)` — resolve the caller; quest must
  exist (`ErrQuestNotFound`) and be available (`ErrQuestTaken`); the caller's
  slot must be empty — personal quest or party quest (`ErrQuestSlotFull`). If
  the caller is in a party → `holderParty = partyID`, else `holderEntity`.
  Announce: `"alice took quest #2: Scout the far shore"` (party:
  `"alice's party took …"`).
- `QuestAbandon(token) (announce, err)` — the caller's active quest (personal,
  or their party's) goes **back to available with progress reset**
  (`ErrNoActiveQuest` if none). Announce: `"alice abandoned quest #2 …"`.

**Progress & completion hooks** (inside existing resolution, holding `w.mu`):
- **Kill ticks** at the bubble kill-award site (`resolveBubbleTurnLocked`):
  after the XP award loop, for each **distinct active kill quest held by a
  surviving player in the bubble** (their personal quest or their party's),
  `progress += killed` — once per quest per turn, NOT once per member (a party
  of 3 in one fight ticks its shared quest once). Cap at `targetN`.
- **Reach check** each world/bubble turn after movement: an active reach quest
  completes when **any holder** (the owner, or any current member of the
  holding party) stands on `goalHex`.
- **Completion** (either kind): `state = completed`; every current holder gets
  `rewardXP` added to `e.xp` (the human +XP% passive applies — it's "XP
  earned"; `syncMaxHPLocked` after, like the kill award); a `system` chat
  announcement (`"Quest complete: Cull the pack — alice, bob each gain 30 XP"`)
  published via a **server-side hook** (see wiring below). Completed quests
  stay completed (the board depletes; fine for a playtest world).

**Membership transitions:**
- `PartyAccept` (joining a party): if the accepter holds a **personal** quest,
  abandon it back to the board first (its own announcement).
- `leavePartyLocked` dissolving a party (<2 members): the party's quest goes
  back to available, progress reset. (A member leaving a party that stays ≥2
  just exits — the quest stays with the party.)
- Disconnect sweep: sweeping a personal-quest holder returns their quest to the
  board. (Party quests ride the dissolve rule via the sweep's leaveParty.)

**Chat announcements from the game layer:** completion/auto-abandon happen
inside turn resolution, where the World has no broker. `NewWorld` gains an
optional announce func — `w.announce func(sender, text string)` set via
`World.SetAnnounce(fn)` from `app.go` (`func(s, t string) { chatBroker.Publish(s, t) }`)
— called (never nil-checked at call sites; default no-op) for system quest
events. Command announcements (take/abandon via HTTP) keep the existing
return-string → publish path.

## Wire (`internal/protocol`) — regen `protocol.gen.ts`

```go
type QuestState string
const (QuestAvailable QuestState = "available"; QuestTaken QuestState = "taken"; QuestCompleted QuestState = "completed")

// QuestView is one quest on the board, as the client sees it.
type QuestView struct {
	ID       int64      `json:"id"`
	Name     string     `json:"name"`
	Kind     string     `json:"kind"`     // "kill" | "reach"
	TargetN  int        `json:"targetN"`  // kill quests
	GoalHex  Hex        `json:"goalHex"`  // reach quests
	Progress int        `json:"progress"`
	RewardXP int        `json:"rewardXp"`
	State    QuestState `json:"state"`
	// Holder: at most one is non-zero.
	HolderEntityID int64 `json:"holderEntityId"`
	HolderPartyID  int64 `json:"holderPartyId"`
}
```

`TurnEvent.Quests []QuestView` — the whole board (~6 entries) on every bundle
(full-snapshot philosophy; the client filters). No new SSE events, no new
request types (commands ride `POST /api/chat`).

## Commands (`internal/server/chat.go`)

Two new verbs in `routeChatCommand`'s switch (before the default), both
`system`-sender like the party verbs:
- `quest` → `deps.World.QuestTake(req.Token, id)` — `rest` parses as an int
  (`strconv.ParseInt`; bad/missing → a readable 422).
- `abandon` → `deps.World.QuestAbandon(req.Token)`.

## Client (`client/`)

**`<QuestPanel>`** — third SolidJS component (`client/src/quest/store.ts` +
`QuestPanel.tsx`, mirroring chat/roster):
- **My quest** (mine or my party's): name, live progress (`3/5` for kill,
  `reach (9, -4)` for reach), and the **XP reward**. On completion it leaves
  the panel (state flips) — and the **stats HUD visibly jumps** (existing
  `Lv X · Y/Z XP` element; no new work needed beyond asserting it).
- **The board**: available quests with id, name, objective, reward — so
  `/quest <id>` has something to read. Completed quests show struck-through or
  drop off (keep it simple: show available + mine only).
- Mounted into `#quest-root` (same pointer-events discipline; placed to not
  collide with chat (bottom-left) or roster (top-right) — e.g. left side under
  the HUD).
- Store updated each turn from `TurnEvent.Quests` + my entity/party id.
- `window.game.quests` (the board) + `window.game.quest` (my active quest view
  or null) for e2e.

## Decisions (reasoned; flag if you disagree)

1. **The whole board rides the bundle** (~6 rows) — full-snapshot philosophy,
   trivially resyncable, no request/response channel needed; `/quests` command
   unnecessary (the panel shows it).
2. **Kill ticks use the bubble-presence rule** — same as XP: a kill in your
   fight counts for your (or your party's) kill quest; once per quest per turn.
3. **Board depletes** (completed stays completed, monsters don't respawn) —
   repeatable/respawning quests arrive with continuous monster spawning, later.
4. **Announce hook (`SetAnnounce`)** rather than handing the World the broker —
   keeps `internal/game` free of a chat dependency (one function seam).
5. **Human +XP% applies to quest rewards** — it's "XP earned" (plan §0 wording).
6. **Quest names/templates are constants** — no text generation.

## Out of scope (later)

- Player-authored quests, NPCs/quest-givers, quest chains, repeatable or
  regenerating quests, gear/item rewards, per-member partial rewards,
  goal-hex map markers (pairs with the §6 selected-path preview feel item),
  reserving the `system` sender name.

## Tests

- **Unit (`internal/game`)**: generation — same seed → identical board,
  6 quests, reach goals inside the reachable region ≥ 8 from origin, kill
  counts 2–4; take/one-slot/not-found/taken/abandon sentinels; kill quest
  ticks on a bubble kill (once per quest per turn, party of 2 ticks once) and
  completes with XP to all current members (late joiner present at completion
  gets full reward); reach quest completes when a holder stands on the goal;
  join-party abandons a personal quest; dissolve/sweep returns quests with
  progress reset; snapshot `Quests` mirrors state.
- **Integration**: `/quest <id>` take → announcement + `TurnEvent.Quests` shows
  taken-by-me; kill progress over HTTP (deterministic monster placement, like
  the XP tests) → completion announcement + **XP jump on the wire**;
  `/quest 999` → 422; `/abandon` with none → 422.
- **e2e (`client/e2e/quests.spec.ts`)**: take a **reach** quest via
  `window.game.sendChat("/quest N")` (pick the reach quest from
  `window.game.quests`), walk to its goal hex (tapHex), assert the panel
  showed progress then completion **and `window.game.xp` + the stats HUD text
  jumped by the reward**. Own Playwright server.

## Risks

- **Determinism**: board generation from seed only; kill ticks/reach checks
  inside the locked turn — no new RNG draws in combat (nothing perturbs the
  seeded combat tests).
- **Double-ticking**: a party fight must tick the shared quest once per turn,
  not per member — the "distinct quests held by surviving bubble players" set
  is the guard; unit-tested with a party of 2.
- **Announce hook lifecycle**: `SetAnnounce` must be set before `Run` starts
  (app.go does both); default is a no-op so tests without chat still pass.
- **Reach goals must be reachable**: drawn from `reachableWalkable` — the same
  guarantee spawns rely on.
- **e2e walk length**: reach goals ≥ 8 hexes from origin means several turns at
  test cadence — keep `TURN_INTERVAL` small (the e2e servers already run at ms
  cadence) and use `waitForFunction` on completion.
