# Milestone 6.2 — Monsters & HP: Design

*Status: DRAFT for review (2026-07-08). Third milestone-6 slice. Builds on 6.1
(#13) — needs its phased `resolveTurn` and the world `seed`. Rebase this branch
onto `main` once #13 merges.*

## Goal

Introduce the first **hostile entity kind** — monsters — plus **HP** on every
entity, monster **spawning**, and **minimal monster AI** (step toward the
nearest player). This is the thing to fight: the prerequisite for combat (6.3).
No damage, attacks, or bump-to-attack yet.

## Design decisions (please review these)

### 1. Entity kind — `Kind` on the wire
`protocol.Entity` gains `Kind string` with constants `EntityPlayer = "player"`
and `EntityMonster = "monster"`. The server `entity` gains a `kind` field.
Players (via `Join`) are `"player"`; spawned monsters are `"monster"`. The
client colours monsters differently. (A string kind, not a bool, so NPCs/other
kinds can be added later.)

### 2. HP on every entity, exposed on the wire
`protocol.Entity` gains `HP int` and `MaxHP int`. Players and monsters both get
HP at creation (`PlayerMaxHP`, `MonsterMaxHP` — new `protocol` constants).
**HP is stored but never changes in 6.2** — nothing deals damage until 6.3.
It's on the wire now so the client can render health bars when combat lands,
and so the model is in place for 6.3.

### 3. Spawning — fixed count at server start, seeded, **default 0**
- `World.SpawnMonsters(n int)` spawns `n` monsters at random **walkable** hexes,
  chosen with the world `seed` (from 6.1) so a given seed is reproducible.
  `NewWorld`'s signature is unchanged (tests keep calling `NewWorld(interval, ticks)`).
- The server wiring (`cmd/rogue/app`) reads a new `MONSTER_COUNT` env
  (`internal/config`) and calls `world.SpawnMonsters(cfg.MonsterCount)` at
  startup.
- **`MONSTER_COUNT` defaults to `0`.** Rationale: every existing integration and
  e2e test spins up a world without setting it; a non-zero default would put
  random monsters into those worlds and destabilise them (moving monsters
  churning the board, polluting occupancy/entity-count assertions). Default 0
  keeps the whole existing suite untouched; monsters are opt-in via the env, and
  a production/dev default can be raised later once combat exists. **Flag for
  review:** if you'd rather ship a non-zero default now, we'd instead harden the
  existing tests — more work, more churn.
- Continuous/over-time spawning and spawn-distance rules (keep monsters away
  from players at start) are out of scope — a later tuning pass.

### 4. Minimal AI — step toward nearest player, stop adjacent
At the **start of each turn** (a `thinkMonstersLocked` step at the top of
`resolveTurn`, before the phased move phase), each monster:
- finds the **nearest player** by hex distance (ties → lowest id, deterministic);
- `Pathfind`s toward that player and sets its `path` to a **single step** along
  the route — *unless* the only remaining step is onto the player's own hex
  (i.e. it's already adjacent), in which case it **waits**. Monsters approach and
  gather around players but never enter a player's hex, because moving onto a
  hostile is an *attack* — and attacks are 6.3.
- monsters with no players in the world idle.

Because monster movement is just a `path`, it resolves in the **same phased move
phase** as players (6.1) — no separate mover logic.

### 5. Client — render monsters distinctly
`entities.ts` colours entities by `Kind`: monsters get a distinct colour (e.g. a
hostile red) vs the existing self-blue / other-gold for players. `window.game`
exposes a `monsters` count (entities whose kind is monster) for e2e assertions.
Health bars are **not** rendered in 6.2 (HP is on the wire but no combat changes
it yet) — deferred to 6.3.

## Explicitly deferred (later slices)

- **Combat**: damage, attacks, bump-to-attack, HP→0/death — 6.3.
- **"Hostiles never share a hex"** (plan §5): in 6.2 monsters can still stack
  (with each other; the AI keeps them out of *player* hexes). The proper
  bump-to-attack + hostile-occupancy rule lands with combat in 6.3.
- **Time bubbles** (mutual-LOS trigger, freeze) — 6.4.
- **Monster variety / types / loot** — later.
- **Health-bar rendering** — 6.3.

## Tests

- **Unit (`internal/game`):** `SpawnMonsters(n)` creates exactly `n` monsters on
  walkable hexes with `kind == monster` and `HP == MaxHP`; the same seed spawns
  them on the same hexes (reproducible). AI: a monster two+ hexes from a player
  steps one hex *closer* after a resolve; a monster adjacent to the (only)
  player does **not** move onto it; with two players it targets the nearer.
- **Integration (`test/integration`):** a server started with `MONSTER_COUNT>0`
  (via a helper that threads the config) exposes monster entities in the turn
  bundle with `kind`/`HP`, and a monster's hex changes over turns as it hunts a
  joined player.
- **e2e (`client/e2e`):** set `MONSTER_COUNT` in the Playwright server; assert
  `window.game.monsters > 0` and that a monster entity is rendered. **Confirm
  the full existing e2e suite stays green with monsters present** (the `make e2e`
  gate is the check; if a few monsters destabilise a spec, harden that spec or
  drop the e2e monster count).

## Risks

- **Existing-test contamination:** mitigated by `MONSTER_COUNT=0` default so
  `NewWorld`/`startServer` worlds have no monsters unless explicitly asked.
- **e2e shared server:** the one Playwright server is shared across specs;
  enabling monsters there affects every spec. Keep the e2e monster count small
  and verify the whole suite.
- **AI targeting a player's hex:** guard against stepping onto a player (the
  "stop adjacent" rule) — without it, a monster would stack onto a player before
  combat exists.
- **Depends on 6.1 (#13):** the world `seed` (spawn reproducibility) and the
  phased `resolveTurn` (where `thinkMonstersLocked` is invoked) come from 6.1 —
  rebase onto `main` after #13 merges before building.
