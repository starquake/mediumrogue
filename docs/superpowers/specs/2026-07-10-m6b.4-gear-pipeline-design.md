# Milestone 6b.4 — Gear & the Modifier Pipeline: Design

*Scope decided 2026-07-10: the **full gear slice** — the combat modifier
pipeline, species passives migrated onto it, items as data, ground drops,
inventory, and the equip/swap UI — one slice, one PR. Companion doc for the
content vocabulary: `docs/rule-based-content-design.md`. Architecture
decision record: the `combat-modifier-pipeline` memory / plan §8 6b note.*

## Goal

Replace the per-trait combat branches (`critMultiplier`, `applyDR`, the
inline human-XP multiply) with a **modifier pipeline**: combat exposes
events; species and gear carry **rule cards** (pure serializable data) that
transform values at those events. Gear becomes the second modifier source:
items are data definitions, monsters drop them on the ground, players pick
them up by walking in, own many, and equip one per slot. After this slice,
adding an item or passive = adding a data literal, not editing combat code.

Player-visible result: monsters sometimes drop loot; you see it on the map,
walk over it to grab it, open a gear panel to equip it, and the numbers it
promises actually happen in combat.

## The pipeline (`internal/game/rules.go`)

**Events implemented this slice** (each a call site where a value is folded
through applicable cards):

| Event | Value | Call sites today |
|---|---|---|
| `deal-damage` | outgoing damage, per (attacker, victim) pair | `attackLocked`, `resolveBowLocked`, `resolveAoELocked` |
| `take-damage` | incoming damage, victim side | same three sites (after deal-damage) |
| `earn-XP` | one player's XP award for a kill | kill award in `resolveBubbleTurnLocked` |

`attack-roll` and `on-kill` from the content guide are **documented, not
implemented** — no card needs them yet (elf crit models cleanly as a
chance-conditioned damage multiplier). The vocabulary is extensible by
design; the backlog item "monster aggro radius via the pipeline" would be
another future event (`aggro-range`), evaluated from monster think code.

**Rule card** — pure data, no closures (the §7 persistence prerequisite):

```go
type ruleCard struct {
    event string      // evDealDamage | evTakeDamage | evEarnXP
    when  []condition // all must hold (AND); empty = always
    then  effect      // one effect per card
}
type condition struct{ kind string; pct, n int } // chance | targetHPBelowPct | targetHPFull | allyInBubble | targetAdjacent
type effect struct{ kind string; n int }         // add (n may be negative) | mulPct
```

**Evaluation** (`applyRules(event, base, ctx) int`):

1. Gather cards from the entity's sources in **stable order**: species cards,
   then the *acting* item's cards (deal-damage: the weapon swung; take-damage:
   victim's close item then ranged item; earn-XP: species only for now).
2. Filter by event, evaluate conditions against a context struct
   (attacker, victim, distance, bubble-mate count, rng). `chance` conditions
   consume the turn's seeded rng **in card order** — deterministic.
3. Arithmetic in fixed phases: sum all `add` effects onto the base, then
   apply all `mulPct` in order, then the **event-level clamp**: `take-damage`
   floors at 1 (a landed hit always costs something), `earn-XP` floors at 0.
   Card order can never change the result within a phase — no stacking bugs.

**Species become the first rule set** (content data, constants stay in
`internal/protocol` for tuning):

- human: `{earn-XP, always, mulPct(100+HumanXPBonusPercent)}`
- elf: `{deal-damage, chance(ElfCritChancePercent), mulPct(100*ElfCritMultiplier)}`
- dwarf: `{take-damage, always, add(-DwarfDamageReduction)}` (+ the global floor-1)

`critMultiplier` / `applyDR` / the inline human multiply are **deleted**.
Semantic change accepted (no-backward-compat): elf crit now rolls **per
victim** in an AoE rather than once per attack, and rng consumption order
changes — affected tests are updated, not preserved.

## Items as data (`internal/game/content.go`)

```go
type itemDef struct {
    id, name, slot, class string // slot: close|ranged; class: fighter|rogue|mage
    damage, rangeHex, aoeRadius int
    rules      []ruleCard
    desc       string // human-readable rule text for the gear panel (authored, not generated)
    dropWeight int    // 0 = never drops (class defaults)
}
```

Registry: ordered slice + map by id, validated at init (unique ids, valid
slots/classes/events). **The old `protocol` weapon-damage constants move
into the defs** (single source: content); level-scaling constants
(`DamagePerLevel`, `HPPerLevel`) stay in protocol.

**Class defaults** (dropWeight 0), replacing `class.go`'s hardcoded weapons
and keeping the **live balance numbers** (the current protocol constants):
`iron-sword` (fighter close, 4), `dagger` (rogue close, 7), `shortbow`
(rogue ranged, 6, range 4), `oak-staff` (mage close, 2), `ember-focus`
(mage ranged, 4, range 4, AoE 1). Fists stay a built-in fallback for an
empty close slot; monsters keep their flat claws (`MonsterAttackDamage`).

**Starter drop set** (guide-card style; balanced relative to the defaults —
lower base, situational spike):

| Item | Class/slot | Base | Rule | Weight |
|---|---|---|---|---|
| `butchers-cleaver` | fighter close | dmg 3 | deal-damage: if target below 50% HP, +3 | 4 |
| `iron-warhammer` | fighter close | dmg 6 | none — flat upgrade, rare | 1 |
| `venom-fang` | rogue close | dmg 5 | deal-damage: if target at full HP, +4 | 4 |
| `pack-bow` | rogue ranged | dmg 5, rng 4 | deal-damage: if an ally shares the bubble, +3 | 4 |
| `ember-staff` | mage ranged | dmg 3, rng 4, AoE 1 | deal-damage: if target adjacent, mulPct(200) | 4 |

## Inventory, slots, equip

- Entity gains `items []itemInstance{id int64, defID string}` plus
  `closeSlot, rangedSlot int64` (instance ids; 0 = empty). Instance ids come
  from the world's entity-id counter (shared sequence, unique everywhere).
- **Join:** class-default instances are created, owned, and equipped.
- **Equip** — new intent kind `equip` with `itemId`:
  - Validation: instance owned by you, def's class matches yours, def's slot
    decides which slot it fills (swap replaces; no bare unequip — YAGNI).
  - **Outside a bubble: immediate and free** (applied directly under the
    world lock, no turn consumed).
  - **Inside a bubble: it is your action for the turn** — commits your
    intent (clearing path/attackTarget), resolves at the start of resolution
    (before moves and attacks: the newly equipped weapon is live this turn,
    but you gave up moving/attacking for it).
- No inventory cap this slice (panel scrolls).
- **Gear survives death** — inventory and equipped slots persist through
  respawn. (Plan §9's corpse-run question stays open.)
- Monsters own no items; their melee stays the fists profile. Monster hits
  still run the pipeline (victim-side dwarf/gear rules apply).

## Ground drops & pickup

- World gains `groundItems map[protocol.Hex][]groundItem{instanceID, defID}`.
- **Drop roll** in `resolveDeathsLocked`, monster deaths only:
  `DropChancePercent` (protocol constant, first draft 30) then a
  dropWeight-weighted pick over all defs with `dropWeight > 0`, using the
  turn's seeded rng. The item spawns on the monster's death hex (class
  restriction does not gate the roll — loot for other classes is party loot).
- **Pickup is automatic**: after the move phase, every entity standing on a
  hex with ground items picks them all up (players only; monsters ignore
  loot). If several players enter together, the lowest entity id collects —
  deterministic. Announce via the existing `SetAnnounce` chat hook
  ("Anna picked up Butcher's Cleaver").
- Ground items persist until picked up (no despawn this slice).

## Wire (`internal/protocol`) — regen `protocol.gen.ts`

- `Entity` += `Items []ItemView` (id, defId, name, slot, damage, range,
  aoeRadius, desc, equipped) — broadcast for all players (15 friends; no
  privacy concern, simplest).
- `TurnEvent` += `GroundItems []GroundItemView` (hex, defId, name).
- `IntentRequest`: `Kind` gains `"equip"`; new field `ItemID int64`.
  Equip requests outside a bubble return 200 and apply instantly; inside a
  bubble they commit as the turn intent.
- `make protocol`; contract test extended with an inventory-carrying bundle.

## Client (`client/`)

- **Ground-item rendering:** a small loot glyph per occupied hex (distinct
  color/shape, under entities). No sprite art — a Pixi-drawn marker, same
  idiom as existing entity markers.
- **`<GearPanel>` (SolidJS):** my inventory grouped by slot; each row shows
  name, stats, desc, an "equipped" badge; click → POST equip intent. Server
  is authoritative on validity; the panel just reflects the next snapshot.
- `window.game` += `inventory`, `equipped`, `groundItems` for Playwright.

## Decisions (reasoned; flag if you disagree)

- **Pipeline lives in package `game`** (unexported, next to its one
  consumer) — a separate package would force exporting entity internals.
- **Crit = chance-conditioned multiplier**, no separate attack-roll event
  until something needs to-hit rolls. Fewer moving parts, same numbers.
- **Item numbers live in content defs, not protocol constants** — the wire
  carries display stats, so the client never needs the numbers at compile
  time. Species percentages stay protocol constants (they're tuning knobs
  referenced by cards).
- **Per-victim crit rolls in AoE** — the pipeline evaluates per pair; the
  old roll-once behavior was an artifact, not a design decision.
- **Equip-before-move-phase** rather than a fourth resolution phase: the
  swap is your whole turn, so it can't interact with movement conflicts.

## Out of scope (later)

Buffs/status effects and durations; `attack-roll`/`on-kill`/`aggro-range`
events; armor/trinket slots; inventory cap; item despawn; drop-on-death
(corpse runs); loot tables per monster kind (one global table now); the
analytics trace of per-modifier contributions (milestone 12); SQLite (§7
path — cards are already pure data, which is the prerequisite).

## Tests

- **Unit (`internal/game`):** pipeline fold order (adds→muls→clamp),
  condition evaluation incl. rng determinism with a fixed seed, species
  cards reproduce exactly the old helpers' numbers, registry validation,
  weighted drop pick, pickup determinism, equip validation (wrong class,
  not owned, in-bubble turn consumption), gear-survives-death.
- **Integration (`test/integration`):** equip over real HTTP (outside
  bubble = instant; inside = consumes the turn), kill→drop→walk→pickup
  loop, inventory on the wire.
- **e2e (`client/e2e`):** join, see a ground item, walk onto it, open the
  gear panel, equip, assert `window.game.equipped` flips.
- **Contract test:** bundle with items + ground items through Go encoder +
  TS decoder.

## Risks

- **Widest wire change since 6b.2** — `Entity` grows; keep `ItemView` flat
  and tygo-friendly.
- **RNG stream changes** ripple through seeded tests — budgeted as test
  updates, not fought.
- **Equip-in-bubble interacts with readiness gating** (`allPlayersReadyLocked`)
  — the equip intent must mark the player ready exactly like move/attack.
- **Panel is the fourth SolidJS panel** — follow the Chat/Roster/Quest
  panel pattern verbatim; no new UI machinery.
