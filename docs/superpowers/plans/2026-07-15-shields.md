# Shields Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Shields v1 (#90, S4 of #55): a 9th `shield` item type that equips
off-hand only and carries a flat `take-damage −N` rule card — two items
(Wooden Buckler −1 common, Iron Kite Shield −2 rare), drop-only. No pipeline
changes, no snapshot bump.

**Architecture:** Content-and-placement first (Task 1: protocol const,
taxonomy, eviction rule, the two defs, all combat/placement tests — nothing
seeded moves because shield cards consume no rng); drop tables isolated
second (Task 2 — the only task where pinned seeds can shift); client mapping
third; docs last. All on the draft PR branch (`docs/shields-spec`) — spec,
plan, and build merge together (maintainer's milestone-slice workflow).

**Tech Stack:** Go server (`internal/protocol`, `internal/game`), tygo
regen, TS client (`client/src/gear`). No new env vars, no e2e changes.

**Spec:** `docs/superpowers/specs/2026-07-15-shields-design.md` (approved on
the draft PR).

## Global Constraints

- Determinism: shield cards carry no `chance` condition — combat rng must
  not move. Only Task 2 (drop tables) may shift pinned seeds; re-derive,
  never weaken (protocol below).
- Suite green at every task commit: `set -o pipefail && make check 2>&1 | tail -15`
  (go may be at `/usr/local/go/bin/go`; cd to the repo root first).
- One draft PR: commit and push to `docs/shields-spec`; do NOT open new PRs
  or merge; the PR is marked ready only in Task 4. Merging stays label-gated
  (`ready to merge` is the maintainer's).
- Go tests use the `got, want` inline style (`.claude/rules/go-style.md`).
- `internal/protocol` consts change → run `make protocol` and commit the
  regenerated `client/src/protocol.gen.ts` in the SAME commit (drift gate).

## Re-derivation protocol (Task 2 only)

Appending a drop row changes that table's total weight, so the same seed can
draw a different item (`pickDrop`'s `IntN(totalWeight)`). When a pinned test
fails: confirm the new value is a legal draw from that kind's table, then
re-hunt/update the pin the way `drops_test.go`'s const block documents
(scan seeds, pick the first that produces the pinned scenario) and comment
`re-derived: #90 shield rows changed the table weights`. Never widen an
assertion.

c) `inventory.go` — `equipItemLocked`, between `equipValidate` and the
toggle-off check, the shield eviction branch:

```go
	// A shield equips into the off-hand (slotForType), but a two-handed
	// weapon in main LOCKS that slot — equipping the shield evicts the
	// two-hander to the backpack first, room-checked BEFORE any state
	// change (mirroring equipWeaponLocked's polite failure). A two-hander
	// in main implies an empty off-hand (equipWeaponLocked's invariant),
	// so eviction is the only prerequisite. #90.
	if def.itemType == protocol.ItemTypeShield && heldSlotOf(e, inst.id) == "" {
		if main := e.equippedDefIn(protocol.SlotMainHand); main != nil && main.twoHanded {
			if e.freeBackpackIndex() < 0 {
				return ErrBackpackFull
			}

			e.toggleEquip(e.equipped[protocol.SlotMainHand], protocol.SlotMainHand)
		}
	}
```

CAREFUL: `heldSlotOf` is the hands-only helper — a shield already in
off-hand means this equip is a toggle-OFF; the eviction must not fire then.
(A shield can only ever occupy off-hand, so `heldSlotOf` == "" is exactly
"not currently equipped".) The existing toggle-off + `toggleEquip` tail of
the function then handles everything else, including swapping out another
shield.

d) `items.go` — `validateItemDefs` tightening (new helper
`validateItemCombatStats`, called next to `validateItemHeal`):

```go
// validateItemCombatStats panics if a non-weapon def sets damage, rangeHex,
// or aoeRadius — only a weapon fires as a hit, so combat stats on a shield
// or armor def are authoring mistakes (a shield's −N lives in its rule
// card, not a damage field). #90.
```
